## 1 分钟项目概述

我这个项目是一个中小型企业库存管理系统，主要解决传统仓库用 Excel 或人工台账管理库存时，容易出现库存变化难追踪、审核责任不清楚、库存预警滞后、不同岗位权限边界不明确的问题。

系统里有商品管理、入库管理、出库管理、库存盘点、库存流水、库存预警、用户权限和操作日志几个模块。业务流程是：先维护商品主数据，然后创建入库单或出库单，单据审核通过后才真正修改库存，同时生成库存流水；如果库存低于或高于安全库存范围，系统会生成预警，后续可以记录处理结果。

后端我用了 Spring Boot、MyBatis-Plus、Spring Security 和 JWT；前端用了 Vue 3、Element Plus、Pinia 和 ECharts；数据库是 MySQL。为了让项目更容易运行，我还加了 Docker Compose，一条命令可以启动 MySQL、后端和前端，并自动导入演示数据。

## 3 分钟项目讲解

### 开场

这个项目我想重点讲三个点：业务闭环、库存一致性、权限和部署。

第一，业务闭环方面，我没有只做商品的增删改查，而是围绕库存变化做了一套流程。商品是基础数据，入库和出库是库存变化的来源，审核通过之后才会修改库存。每一次库存变化都会生成库存流水，后面如果库存低于安全库存下限，或者高于安全库存上限，会生成库存预警。

### 入库和出库

入库模块里，用户可以创建入库单，单据初始是待审核状态。只有审核通过后，系统才会增加对应商品的库存，并记录审核人、审核时间和关联流水。

出库模块和入库类似，但出库多了一个库存校验。审核出库单的时候，系统会判断当前库存是否足够，如果库存不足就不能审核通过。审核成功后才扣减库存，并生成出库流水。

我这样设计是因为真实业务里，创建单据不等于库存已经变化。只有审核通过，才代表这个业务动作正式生效。

### 库存流水

库存流水是我觉得这个项目比较关键的地方。因为如果只是改商品表里的当前库存，后面很难知道库存为什么变了。

所以我在库存变化时记录了变更类型、变更数量、变更前库存、变更后库存、关联单据和操作人。这样后面排查问题时，可以根据流水追踪库存变化来源。

### 库存预警

库存预警是根据商品表里的安全库存上下限生成的。比如库存低于下限，就生成低库存预警；高于上限，就生成高库存预警。

预警不是只展示一条提醒，还可以记录处理方式、责任人、处理说明和跟进时间。这样可以把库存异常从“看到问题”推进到“处理问题”。

### 权限控制

权限这块我用了 JWT + Spring Security + RBAC。用户登录成功后，后端返回 token，前端保存 token，后续请求带上 token。后端通过过滤器解析 token，识别当前用户。

权限模型是用户、角色、权限三层。前端根据权限控制菜单和路由展示，后端接口也会做权限校验。我理解前端控制更多是提升体验，真正的安全边界还是要放在后端。

### Docker Compose

为了让项目能方便展示，我加了 Docker Compose。它会同时启动 MySQL、后端和前端。MySQL 容器初始化时会执行建表脚本、基础数据脚本和演示数据脚本。

这样面试官或者其他同学拿到项目后，不需要手动建库、导 SQL、分别启动前后端，一条 `docker compose up --build` 就能看到完整系统。

### 收尾

所以这个项目对我来说，重点不是做了多少页面，而是把库存管理里的单据审核、库存一致性、流水追踪、预警处理、权限控制和部署交付串成了一个完整闭环。

## 常见追问回答

### 1. 为什么只准备了一个项目？

我目前把这个项目作为主项目来准备。因为我觉得实习面试阶段，项目数量不是最重要的，关键是能不能把一个项目讲深、跑通、讲清楚。

这个 ERP 项目覆盖了后端分层、数据库表设计、登录权限、事务、库存流水、预警、前后端分离和 Docker 部署，比临时堆几个简单 CRUD 项目更能体现我的学习深度。后面如果时间允许，我会再补一个小项目，但目前我想先把这个主项目打磨扎实。

### 2. 这个项目和普通 CRUD 有什么区别？

普通 CRUD 主要是对单表做新增、删除、修改、查询。这个项目除了基础维护以外，还有业务流程。

比如出库不是简单修改库存，而是先创建出库单，审核时校验库存，审核通过后扣减库存，再生成库存流水。库存预警也不是单纯列表，而是根据安全库存规则生成，并且可以记录处理结果。权限也不是只做登录，而是有角色和权限控制。

### 3. 库存怎么保证不乱？

我主要从三个方面处理。

第一，创建单据不直接改库存，只有审核通过才修改库存。

第二，库存修改和库存流水记录放在同一个事务里，避免库存改了但流水没记录，或者流水记录了库存没改。

第三，出库审核时会校验库存是否充足，库存不足就不允许审核通过。

如果继续优化，我会增加更多并发扣减测试，并结合数据库行锁或乐观锁进一步强化库存一致性。

### 4. JWT 登录流程是什么？

用户登录时，前端把用户名和密码提交给后端。后端校验成功后，根据用户信息生成 JWT token 返回给前端。

前端保存 token，后续请求把 token 放到请求头里。后端的认证过滤器会解析 token，判断 token 是否有效，再把用户身份放到当前请求上下文中。这样后面的接口就能知道当前是谁在操作。

### 5. 权限是怎么做的？

权限模型是用户、角色、权限。一个用户可以绑定角色，角色再绑定权限。

前端根据当前用户权限控制菜单和路由显示，避免用户看到自己不能操作的功能。后端接口也会做权限校验，防止用户绕过前端直接调用接口。

我的理解是，前端权限主要是用户体验，后端权限才是安全保障。

### 6. 为什么要有库存流水？

如果只有商品表里的当前库存，那么库存从 100 变成 80，只能知道结果，不知道原因。

库存流水记录了变更类型、数量、变更前库存、变更后库存、关联单据和操作人。这样可以追踪这次变化是入库、出库还是盘点调整产生的，也方便后续审计和排查问题。

### 7. Docker Compose 解决了什么问题？

它主要解决运行门槛问题。

以前别人拿到项目，需要自己安装 MySQL、建库、导 SQL、启动后端、启动前端。现在通过 Docker Compose，可以一条命令启动 MySQL、后端和前端，并自动导入演示数据。

这对面试展示也有帮助，因为项目不是只能在我电脑上跑，而是更容易被别人复现。

### 8. 项目里遇到过什么问题？

我遇到过几个比较典型的问题。

一个是前后端联调时的跨域和 token 携带问题，后来通过统一封装 Axios 请求和后端 CORS 配置解决。

另一个是 Docker Compose 一键启动时，数据库初始化和后端启动顺序的问题。后端如果在 MySQL 还没准备好时启动，就会连接失败。后来我给 MySQL 加了 healthcheck，让后端等待数据库健康后再启动。

还有一个是演示数据问题。如果数据太少，页面看起来像空系统；如果数据不能重复导入，演示前重置会很麻烦。所以我把演示数据脚本设计成可以重复执行，先清理 `DEMO-` 前缀的数据，再重新插入。

### 9. 你觉得项目还有哪些不足？

我觉得还有几个可以继续优化的地方。

第一，可以补充 CI 流程，让 GitHub 自动跑后端测试和前端构建。

第二，可以增加更多并发库存扣减测试，进一步验证库存一致性。

第三，可以完善接口文档和部署说明，比如补 Knife4j 截图、生产环境 Nginx、HTTPS 和数据库备份策略。

第四，前端打包体积还可以通过路由懒加载和分包优化。

### 10. 如果让你入职后维护类似系统，你会怎么做？

我会先熟悉业务流程和数据库表结构，再看核心接口和权限设计。

如果要改功能，我会先确认这个功能会影响哪些表、哪些接口、哪些页面，再小范围修改，并补对应测试。对于库存这种敏感业务，我会特别关注事务、并发和日志记录，避免只改表面页面。

## 练习方式

建议按这个顺序练：

1. 先把 30 秒自我介绍练到自然。
2. 再把 1 分钟项目概述练熟。
3. 然后挑 3 个技术亮点重点练：库存一致性、权限控制、Docker Compose。
4. 最后练常见追问，不需要一字不差，但要能按自己的话讲出来。

最重要的是语气稳一点。实习面试不要求你像高级工程师一样回答，但要让面试官感觉你做过、理解过、愿意继续学。
