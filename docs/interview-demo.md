# 面试演示脚本

这份脚本用于实习面试时快速讲清楚项目价值、技术实现和演示路径。建议控制在 3 到 5 分钟，先讲业务闭环，再讲技术亮点。

## 一句话介绍

这是一个基于 `Spring Boot + Vue 3 + MySQL` 的 ERP 库存管理系统，用来解决中小型企业依赖 Excel 管理库存时出现的库存变化难追踪、审核责任不清晰、预警滞后和权限边界模糊的问题。

## 演示前准备

```bash
docker compose up --build
```

访问地址：

```text
前端：http://localhost:3000
后端：http://localhost:8080
MySQL：localhost:3307
```

默认登录：

```text
admin / admin123
```

如果需要重置演示数据：

```bash
docker compose down -v
docker compose up --build
```

## 3 分钟演示路线

### 1. 登录与首页

打开 `http://localhost:3000`，使用 `admin / admin123` 登录。

讲解重点：

- 首页展示商品总数、库存预警、入库总量、出库总量。
- 图表不是装饰，而是把库存数据汇总成可快速判断的经营视图。
- 演示数据已预置，面试官拉取项目后可以直接看到业务状态。

### 2. 商品管理

进入“商品管理”，展示商品 SKU、分类、价格、当前库存和安全库存上下限。

讲解重点：

- 商品是入库、出库、盘点、预警的基础主数据。
- 安全库存上下限会影响库存预警。
- 支持查询、分页、编辑和导出，符合后台管理系统的常见使用习惯。

### 3. 入库管理

进入“入库管理”，展示入库单列表和审核状态。

讲解重点：

- 入库单审核通过后才会增加库存。
- 审核会记录审核人和审核时间。
- 业务变化会同步生成库存流水，方便追踪来源。

### 4. 出库管理

进入“出库管理”，展示出库单和库存扣减逻辑。

讲解重点：

- 出库审核前会校验库存是否充足。
- 审核通过后扣减库存并生成流水。
- 这一步体现了库存业务中最重要的约束：不能因为重复提交或旧数据覆盖造成库存不一致。

### 5. 库存流水

进入“库存流水账”，展示每次库存变化的变更前库存、变更后库存、关联单据和操作人。

讲解重点：

- 库存流水解决“库存为什么变了”的追踪问题。
- 入库、出库、盘点调整都会进入同一套流水。
- 这个页面是审计和排查问题的重要入口。

### 6. 库存预警

进入“库存预警”，展示低库存和高库存预警。

讲解重点：

- 系统根据商品安全库存上下限生成预警。
- 预警可以记录处理方式、责任人、说明和跟进时间。
- 这让库存管理从事后发现问题变成主动提醒。

### 7. 权限与日志

如果面试时间允许，再展示“用户管理”和“操作日志”。

讲解重点：

- 登录使用 JWT，后端用 Spring Security 校验权限。
- 前端菜单和路由会根据权限控制展示。
- AOP 记录关键操作日志，便于审计。

## 技术亮点讲法

### 库存一致性

入库、出库、盘点审核都在 Service 层事务中完成，库存变更和流水记录一起提交。出库审核会校验库存充足，库存更新逻辑避免旧库存覆盖新库存。

可以这样说：

> 我把库存变化放在审核环节，而不是创建单据时直接改库存，这样能保证业务动作可审核，也能避免草稿单据影响真实库存。

### RBAC 权限控制

系统使用用户、角色、权限三层模型。前端根据权限控制菜单和路由，后端接口再通过 Spring Security 和权限校验兜底。

可以这样说：

> 前端隐藏按钮只能改善体验，真正的安全边界放在后端，所以接口层也做了权限校验。

### 库存流水与操作日志

库存流水记录业务数据变化，操作日志记录用户行为，两者分别解决“数据为什么变了”和“是谁做了什么”的问题。

可以这样说：

> 我没有只做 CRUD，而是补了流水和日志，让系统具备追踪和审计能力。

### Docker Compose 一键启动

项目支持一条命令启动 MySQL、后端和前端。MySQL 初始化时会自动执行建表脚本和演示数据脚本。

可以这样说：

> 为了方便面试官验证，我把数据库、后端、前端都放进 Docker Compose，并预置了演示数据，拉下来后能直接跑起来。

## 常见追问准备

| 问题 | 回答要点 |
| --- | --- |
| 为什么做这个项目？ | 库存管理业务边界清晰，适合展示权限、事务、报表、流水和预警等后端能力。 |
| 和普通 CRUD 有什么区别？ | 有审核流、库存一致性、流水追踪、预警闭环、RBAC 权限和操作日志。 |
| 如何保证库存不乱？ | 审核时才改库存；库存更新、流水记录在同一事务；出库校验库存充足；盘点使用库存快照。 |
| 权限怎么做？ | JWT 登录认证，RBAC 管理角色权限，前端控制菜单和路由，后端接口校验兜底。 |
| 为什么加 Docker Compose？ | 降低运行门槛，让面试官无需手动配置 MySQL、后端和前端即可验证项目。 |
| 下一步会怎么优化？ | 增加 CI、更多并发测试、接口文档截图、生产部署说明和数据库备份策略。 |

## 结束总结

最后可以用这段话收尾：

> 这个项目对我来说不是单纯做页面和 CRUD，而是把库存管理里的单据审核、库存一致性、流水追踪、预警处理和权限控制串成一个完整闭环。它能体现我对 Java 后端分层、数据库建模、事务控制、前后端分离和部署交付的理解。
